<p>The official new <a href="https://modelcontextprotocol.io/specification/2025-03-26/basic/authorization#2-3-1-server-metadata-discovery-headers">revised version</a> supports OAuth2 authentication method, let’s take a look together.</p> <p>First, the protocol requirements:</p> <ul> <li>STDIO not supported, STDIO only supports passing through env</li> <li>SSE/Streamable HTTP should support it (non-mandatory)</li> </ul> <p>MCP follows the OAuth2 standard, mainly involving:</p> <ul> <li><a href="https://datatracker.ietf.org/doc/html/draft-ietf-oauth-v2-1-12"><strong>OAuth 2.1 IETF DRAFT</strong></a></li> <li>OAuth 2.0 Authorization Server Metadata (<a href="https://datatracker.ietf.org/doc/html/rfc8414"><strong>RFC8414</strong></a>)</li> <li>OAuth 2.0 Dynamic Client Registration Protocol (<a href="https://datatracker.ietf.org/doc/html/rfc7591"><strong>RFC7591</strong></a>)</li> </ul> <p>Among them, OAuth2.1 is still in the draft stage and has not yet become an official RFC standard. Here’s a simple summary of OAuth-related RFCs:</p> <table> <thead> <tr> <th>Standard/Draft</th> <th>Type</th> <th>Status</th> <th>Core?</th> <th>Introduction</th> </tr> </thead> <tbody> <tr> <td>RFC 6749</td> <td>OAuth 2.0 Framework</td> <td>✅ Published</td> <td>✅ Yes</td> <td>Defines authorization flows (authorization code, implicit, password, client credentials) and four major roles (client, resource owner, authorization server, resource server)</td> </tr> <tr> <td>RFC 6750</td> <td>Bearer Token Usage</td> <td>✅ Published</td> <td>✅ Yes</td> <td>Describes how to securely use access tokens (Bearer Token) in HTTP</td> </tr> <tr> <td>RFC 7591</td> <td>Dynamic Client Registration</td> <td>✅ Published</td> <td>❌ Extension</td> <td>Allows clients to dynamically register to the authorization server via API</td> </tr> <tr> <td>RFC 8414</td> <td>Authorization Server Metadata</td> <td>✅ Published</td> <td>❌ Extension</td> <td>Provides a .well-known endpoint for exposing server configuration information, supporting auto-discovery</td> </tr> <tr> <td>draft-ietf-oauth-v2-1</td> <td>OAuth 2.1 Draft</td> <td>⏳ In Draft</td> <td>✅ Proposed Core</td> <td>Summarizes and updates OAuth 2.0 core content, merges and replaces RFC 6749 and RFC 6750, eliminates insecure authorization methods (such as implicit authorization) and enforces PKCE</td> </tr> </tbody> </table> <h2 id="oauth-20--21">OAuth 2.0 &amp; 2.1</h2> <h3 id="oauth-20---rfc-6749--rfc-6750">OAuth 2.0 - RFC 6749 &amp; RFC 6750</h3> <p>Let’s briefly go through the OAuth content in RFC 6749, which covers the definition of OAuth2.0, focusing on the important parts:</p> <ol> <li>Register Client, obtain ClientID and ClientSecret</li> <li> <p>Configure on the Client, when authorization is required, the Client (usually the browser) redirects to /authorize to request user authorization</p> <div class="language-bash highlighter-rouge"><div class="highlight"><pre class="highlight"><code>GET /authorize?

The main differences are:</p> <ul> <li>Implicit and Resource Owner Password Credentials methods were removed due to security concerns</li> <li>All clients are required to use PKCE</li> <li>Refresh token rotation is recommended (the old one is discarded when refreshed)</li> </ul> <p>I think the most important obvious difference is the mandatory use of PKCE (Proof Key for Code Exchange), a mechanism designed to prevent authorization code interception and replay, which basically works as follows:</p> <ol> <li>The client randomly generates a string, which is the code_verifier</li> <li>After SHA256 hashing and Base64 encoding the code_verifier, another random string is obtained, which is the code_challenge</li> <li>The client can send the code_challenge and the corresponding hash algorithm code_challenge_method=S256 when making a request</li> <li>After the user agrees to authorize, the server records the code_challenge</li> <li>When the client exchanges the authorization_code for a token, it must include the original code_verifier</li> <li>The authorization service will hash the code_verifier and compare it to confirm whether to accept the request</li> </ol> <p>Let’s look at the code:</p> <div class="language-python highlighter-rouge"><div class="highlight"><pre class="highlight"><code><span class="kn">import</span> <span class="n">secrets</span>

<span class="n">code_verifier_bytes</span> <span class="o">=</span> <span class="n">code_verifier</span><span class="p">.</span><span class="nf">encode</span><span class="p">(</span><span class="sh">'</span><span class="s">utf-8</span><span class="sh">'</span><span class="p">)</span>
<span class="n">sha256_digest</span> <span class="o">=</span> <span class="n">hashlib</span><span class="p">.</span><span class="nf">sha256</span><span class="p">(</span><span class="n">code_verifier_bytes</span><span class="p">).</span><span class="nf">digest</span><span class="p">()</span>

There is actually some controversy here, with some people believing this is not best practice. We can see <a href="https://github.com/modelcontextprotocol/modelcontextprotocol/issues/205">the discussion here</a>. That is, now the MCP Server is more often treated as an OAuth authorization server, which is a burden for MCP Server providers. Most MCP Servers tend to be lightweight or microservice-oriented, and requiring them to integrate the corresponding authentication is undoubtedly a huge effort.</p> <p>Based on this, MCP Gateway is currently developing a client-side authentication system, allowing MCP Gateway to adapt to more scenarios, and various services can also connect to MCP Gateway to quickly adapt to authentication scenarios.</p> <p>If you’re interested in my open-source project, feel free to use, provide feedback, and participate in any contributions:</p> <p>https://github.com/mcp-ecosystem/mcp-gateway</p>
